# TrapTunnel 告警管理平台整体架构设计

## 1. 文档目标

本文档描述以 TrapTunnel `node` 为接入基础设施时，一套理想化但可落地的设备运维告警管理平台应该如何分层、如何流转数据、如何实现高可用，以及在当前仅有 `A-F` 六台入口设备的约束下如何进行现实部署。

本文档的重点不是单个程序如何实现，而是整套系统的职责边界和协作方式。

## 2. 总体定位

整套系统应拆为以下四层：

1. `Node` 接入与传输层
2. `MQ` 事件总线层
3. `Alert` 告警处理层
4. `TDSQL` 状态与配置层

外加一个对外动作层：

5. 通知、工单、北向接口和页面展示层

理想链路如下：

`设备 -> Node -> MQ -> Alert -> TDSQL -> 通知/工单/北向接口`

遗留兼容链路作为旁路存在：

`Node inject -> inManage`

## 3. 各层职责边界

### 3.1 Node 接入与传输层

`node` 是数据面入口，不是告警管理系统本体。

它负责：

- 接收设备发来的 Trap
- 使用 Keepalived VIP 提供设备入口高可用
- 保留原始 Trap 的报文事实
- 将原始数据统一封装成标准 frame
- 在节点之间做 relay 和 egress 分发
- 为遗留系统做本地 inject 兼容
- 对外提供 export 接口

它不负责：

- 告警规则判断
- 重复告警合并
- 告警生命周期管理
- 通知和工单
- 业务配置中心

它的系统角色是：

- 采集代理
- 传输网关
- 协议适配器

### 3.2 MQ 事件总线层

MQ 是采集层和处理层之间的缓冲与解耦层。

它负责：

- 接住 Node 输出的事件流
- 削峰填谷
- 支持多消费者并行处理
- 提供消息重投、补偿和重放能力
- 让 Alert 与 Node 解耦

它不负责：

- 告警规则解释
- 告警聚合
- 告警状态存储

MQ 承载的是事件，不是最终告警。

### 3.3 Alert 告警处理层

Alert 层是真正的告警引擎。

它负责：

- 解析 Trap
- 归一化字段
- 生成业务语义上的 `alarm_key`
- 重复告警合并
- 抑制、去重、压缩、关联
- 生命周期管理
- 生成告警对象
- 写入状态库
- 触发通知、工单和北向同步

它必须是一个集群，而不是多台机器上的多套独立程序。

### 3.4 TDSQL 状态与配置层

TDSQL 是共享状态库和统一配置源。

它负责：

- 保存统一规则配置
- 保存当前活跃告警状态
- 保存历史告警记录
- 保存原始事件索引
- 保存通知和工单关联信息
- 作为 Alert 集群的单一事实源

它不负责：

- 替代 MQ
- 直接接收 Trap
- 替代 Node

### 3.5 动作与展示层

这一层负责：

- 邮件、短信、IM、Webhook
- 工单系统对接
- 页面展示
- 北向 API
- 告警确认、关闭、派单、升级

它应依赖 Alert 层输出的标准化告警对象，而不是直接依赖原始 Trap。

## 4. 数据模型与流转

## 4.1 原始事件

一条原始 Trap 进入系统后，理想上至少应形成一个标准事件对象，包含：

- `event_id`
- `source_node_id`
- `sequence_id`
- `recv_time`
- `raw_ip_packet`
- `udp_src_ip`
- `udp_dst_ip`
- `udp_src_port`
- `udp_dst_port`
- `snmp_version`
- `enterprise_oid`
- `generic_trap`
- `specific_trap`
- `agent_addr`
- `resolved_source_ip`
- `ingress_node`
- `ingress_group`

设计原则：

- 原始事实单独保留
- 解析出的业务字段单独存放
- 不能用修正后的业务字段覆盖原始字段

### 4.2 告警对象

Alert 层在处理事件后，生成业务层的告警对象。

典型字段包括：

- `alarm_key`
- `severity`
- `source_ip`
- `resource_id`
- `alarm_name`
- `first_seen`
- `last_seen`
- `repeat_count`
- `status`
- `latest_event_id`
- `config_version`

### 4.3 完整链路

完整数据流如下：

1. 设备将 Trap 发往某个 VIP
2. 当前持有 VIP 的 Node 接收到 Trap
3. Node 保留原始报文，生成标准 frame
4. Node 将事件输出到 MQ
5. Alert 集群从 MQ 消费
6. Alert 解析事件，生成 `alarm_key`
7. Alert 查询或更新共享状态
8. 若为新告警，则创建告警对象并触发动作
9. 若为短时间重复告警，则更新合并计数和时间窗口
10. 结果写入 TDSQL
11. 页面、通知、工单、北向接口消费告警结果

兼容旁路：

1. Node 在 `inject` 路径复制 payload
2. 若是 SNMPv1 Trap，则按 `agent-addr` 修正注入副本源 IP
3. 将修正后的副本注入本地 `162/UDP`
4. `inManage` 继续以兼容方式接收 Trap

## 5. SNMPv1 agent-addr 的系统意义

当前问题的根源在于：

- 经过代理转发的 SNMPv1 Trap，其 UDP 外层源 IP 往往不是原始设备地址
- 原始设备地址存在于 Trap PDU 的 `agent-addr` 字段中
- `inManage` 无法区分两者，默认按 UDP 外层源 IP 识别设备

因此系统中应明确三个概念：

- `udp_src_ip`
  - 原始网络层源地址
- `agent_addr`
  - SNMPv1 Trap 中声明的设备地址
- `resolved_source_ip`
  - 业务上用于识别设备的地址

设计原则：

- 主链路保留原始 `udp_src_ip`
- 业务侧新增 `resolved_source_ip`
- `inject` 路径才允许用 `agent_addr` 覆盖注入副本源 IP

## 6. 重复告警合并设计

重复告警合并不应该在 Node 或 MQ 层完成，而应在 Alert 层完成。

### 6.1 合并的基本策略

Alert 层应生成一个稳定的 `alarm_key`，常见构成包括：

- `resolved_source_ip`
- `enterprise_oid`
- `generic_trap`
- `specific_trap`
- `resource_id`
- `severity`

若一段时间内出现相同 `alarm_key` 的事件，则视为重复告警。

### 6.2 MQ 配合策略

为降低并发冲突，应让同一个 `alarm_key` 的消息尽量落到同一个 MQ 分区或路由。

效果：

- 同类事件尽量由同一个 Alert 实例顺序处理
- 降低 6 个 Alert 并发更新同一告警状态的概率

### 6.3 状态存储策略

重复告警状态必须存放在共享存储中，而不是存放在单机内存中。

建议使用：

- `alarm_current`
  - 当前活跃告警
- `alarm_event`
  - 原始事件流水
- `alarm_history`
  - 状态变化历史

当重复事件到来时：

- 不新建多条相同活跃告警
- 只更新 `last_seen`
- 递增 `repeat_count`
- 记录最近一次 `event_id`

## 7. Alert 集群如何共享同一套配置

6 个 Alert 实例必须视为同一个集群，不能各自维护私有规则文件。

建议以 TDSQL 作为配置中心，保存：

- 规则定义
- 合并窗口
- 抑制策略
- 通知策略
- 路由策略
- 配置版本号

工作方式：

1. Alert 启动时从 TDSQL 拉取全量配置
2. 在内存中构建只读配置快照
3. 轮询配置版本，或接收配置更新通知
4. 版本变化时整体热切换到新配置

关键要求：

- 所有实例使用同一配置版本
- 日志中记录处理某条告警时使用的配置版本
- 支持配置回滚

## 8. 高可用设计

整套系统应按层考虑高可用，而不是把所有能力混成一个主备概念。

### 8.1 设备入口高可用

- 由 `A/B`、`C/D`、`E/F` 三组 VIP 承担
- 这是接入层的高可用

### 8.2 事件传输高可用

- 由 MQ 集群承担
- MQ 应独立于 VIP 发现机制工作
- Node 应显式配置 broker 列表，不依赖单 VIP

### 8.3 告警处理高可用

- 由多个 Alert 实例共同组成一个消费集群
- 不再采用传统“1 主 5 备”的应用主备方式

### 8.4 数据库高可用

- 由 TDSQL `1主3从 + VIP` 承担
- Alert 通过 TDSQL VIP 访问数据库

## 9. 理想架构与现实部署

### 9.1 理想架构

若机器资源充足，最理想的部署形态是：

- `A-F` 只部署 Node
- 独立 3 台部署 MQ
- 独立 2 到 3 台部署 Alert
- 独立部署 TDSQL 或使用现有数据库集群

这种情况下：

- 采集层
- 事件总线层
- 处理层
- 数据层

能够完全分离。

### 9.2 当前现实约束

当前无法拆出独立机器，因此必须混部。

在当前只有 `A-F` 六台机器，且它们天然分为三组故障域的条件下，推荐部署为：

- `A-F` 全部部署 Node
- `A/C/E` 部署 MQ Broker
- `A-F` 部署 Alert，至少 `B/D/F` 部署主消费实例
- Alert 全部访问 TDSQL VIP

推荐理解为：

- `A/C/E`
  - `Node + MQ + Alert`
- `B/D/F`
  - `Node + Alert`

## 10. 混部场景下的最佳实践

既然无法物理分层隔离，就必须做资源隔离。

建议：

- `Node` 优先级最高
- `MQ` 优先级次高
- `Alert` 优先级最低
- 使用 systemd/cgroups 或容器限制 CPU、内存和 IO
- MQ 数据目录独立
- Node 日志目录与 MQ 数据目录分离
- 避免大日志 IO 拖慢 MQ

系统角色优先级：

1. `Node`
2. `MQ`
3. `Alert`

因为：

- Node 是入口
- MQ 是解耦缓冲
- Alert 属于可横向扩展的处理层

## 11. MQ 失效时的降级思路

首选路径仍然应是：

`Node -> MQ -> Alert -> TDSQL`

但考虑到 MQ 是 3 节点仲裁，可能出现失去多数派的情况，因此需要应急降级设计。

### 11.1 不建议的主降级方案

不建议将“修改所有 Node 配置，使其从写 MQ 改为直写 Alert”作为默认主降级方案。

原因：

- 切换面太大
- 恢复后难以处理重复和乱序
- 容易形成两条主路径并存

### 11.2 更合理的降级方案

更合理的应急策略是：

- Node 始终保留 `export` 能力
- 正常时 Alert 从 MQ 消费
- MQ 故障时，Alert 临时切换为从 Node `export` 订阅
- 恢复后再切回 MQ

该方式的优点：

- Node 侧无需大规模改配置
- 切换面主要集中在 Alert 侧
- 原始事件标识可以复用，便于幂等处理

## 12. TDSQL 中建议的核心表

建议至少包含以下核心表：

- `event_raw`
  - 原始事件流水
  - 主键或唯一键：`event_id`
- `alarm_current`
  - 当前活跃告警
  - 唯一键：`alarm_key`
- `alarm_history`
  - 告警生命周期历史
- `rule_definition`
  - 规则定义
- `policy_definition`
  - 合并、抑制、通知等策略
- `config_version`
  - 当前配置版本及变更记录

关键要求：

- 所有写入都幂等
- 重复消费不会造成状态错乱
- 所有状态变更可追溯

## 13. 设计原则总结

整套系统应坚持以下原则：

1. 原始 Trap 事实保留，不在主链路覆盖
2. `Node` 只负责接入与传输，不负责告警业务
3. `MQ` 只负责事件流，不负责告警状态
4. `Alert` 负责理解和管理告警
5. `TDSQL` 负责共享状态和统一配置
6. 遗留系统兼容能力只放在 `inject` 路径
7. 同类告警尽量路由到同一个处理路径
8. 所有关键写操作都必须幂等

## 14. Alert 技术演进路线

考虑到当前系统仍处于方案收敛和能力建设阶段，Alert 层不必从一开始就做成完全分布式多活集群。更务实的路线是先把业务语义、数据模型和幂等约束做正确，再逐步提升可用性和并行度。

### 14.1 第一阶段：单活处理 + 多备部署

目标：

- 用最小复杂度把告警识别、合并、抑制、生命周期和通知链路跑通

推荐形态：

- 多个 Alert 实例部署在多台机器上
- 但同一时刻只有一个实例真正从 MQ 消费
- 其他实例作为 standby，不参与实际消息处理

数据流：

`Node -> MQ -> Alert(active) -> TDSQL`

特点：

- 开发复杂度最低
- 告警合并逻辑可以按单活思路实现
- 数据库并发冲突问题最少
- 配置变更和问题排查最简单

适用场景：

- 单节点性能足够覆盖全部告警处理量
- 业务规则变化频繁
- 当前阶段优先追求“先把语义做对”

关键要求：

- 即使是单活，也要保持 `event_id` 全局唯一
- 即使是单活，也要设计稳定的 `alarm_key`
- 数据库存储仍需尽量幂等
- 所有实例都应加载同一版本配置

### 14.2 第二阶段：单活处理 + 自动切换

目标：

- 在保持单活处理模型的前提下，减少主实例故障带来的人工切换成本

推荐形态：

- 仍然保持“同一时刻只有一个实例消费”
- 但主备切换由外部机制自动或半自动完成

可选实现：

- 运维开关控制谁是 active
- 基于 Keepalived/VIP 的主备切换
- 基于 leader election 的消费资格控制

特点：

- 业务语义仍然简单
- 不引入多活消费的并发复杂度
- 可显著提升处理层可用性

适用场景：

- 规则语义已基本稳定
- 需要降低单节点故障的恢复时间
- 仍未达到必须水平扩展处理能力的程度

### 14.3 第三阶段：多活消费组

目标：

- 通过多个实例并发消费 MQ，提升吞吐能力和处理冗余

推荐形态：

- 多个 Alert 实例共同组成同一个 consumer group
- 同一条消息只由组内一个实例处理
- 同一 `alarm_key` 的消息尽量路由到同一分区或同一处理路径

数据流：

`Node -> MQ -> Alert Consumer Group -> TDSQL`

特点：

- 吞吐能力更高
- 对单实例故障更不敏感
- 能更充分利用多台机器资源

代价：

- 引入消费重平衡复杂度
- 引入重复消费处理复杂度
- 引入并发写库和共享状态竞争
- 对 `event_id`、`alarm_key`、幂等写入和配置一致性要求更高

适用场景：

- 单活实例已接近性能瓶颈
- 高可用要求提高，不希望存在明显切换窗口
- 规则和数据模型已经稳定

### 14.4 推荐的实施顺序

建议按以下顺序推进：

1. 先实现“单活处理 + 多备部署”
2. 再实现“单活处理 + 自动切换”
3. 最后再升级到“多活消费组”

不建议一开始就直接进入多活消费组，因为这会把业务语义建设和分布式协同复杂度叠加在一起，显著提高开发和排障成本。

### 14.5 各阶段都应提前具备的基础能力

无论当前处于哪个阶段，以下设计最好在第一版就具备：

- `event_id` 全局唯一
- `alarm_key` 稳定可复现
- 数据库写入尽量幂等
- 共享状态落库，不完全依赖本地内存
- 配置中心使用统一版本号
- 配置支持热更新与回滚

这样做的意义在于：

- 即使当前是单活，也不会阻碍未来升级到多活
- 后续技术演进主要是“运行模式升级”，而不是“推翻重写业务模型”

## 15. 结论

理想中的整套告警管理系统，不应再是“6 套边缘告警程序 + 2 套中心接收程序”的拼接式系统，而应演进为：

- `Node` 负责接入和导出
- `MQ` 负责承载事件流
- `Alert` 负责处理和合并告警
- `TDSQL` 负责配置和状态

在这个体系下：

- `inManage` 仍可通过 `inject` 兼容运行
- 新告警系统不再依赖 `162/UDP`
- 重复告警合并和统一配置都可以在集群中稳定实现
- 整体架构更符合长期演进方向和最佳实践
